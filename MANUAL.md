# OpenBAO Genesis Kit Manual

The **OpenBAO Genesis Kit** deploys an OpenBAO cluster using integrated Raft
storage and BPM process management. OpenBAO is an open-source fork of
HashiCorp Vault that provides secure storage of and access to sensitive
credentials used elsewhere in deployments and applications.

Unlike the Vault kit, OpenBAO uses Raft consensus for storage instead of
Consul, eliminating the need for a separate Consul cluster. Peer discovery
is handled via BOSH links and all inter-node communication is secured with
mTLS.

The `safe` CLI is fully API-compatible with OpenBAO, so all operator
workflows (init, seal, unseal, target, auth) work unchanged.

## Version Compatibility

This kit requires Genesis 3.1.0 or higher. It is currently at version 1.0.0.

## Features

The following features are supported by this kit:

- `ocfp` - Uses the config generated by the OCFP scripts that initialize the
  infrastructure, and sets opinionated configurations. This has the following
  effects:

  * sets the network, vm_type and disk_type to unique name specific to this
    environment, so no chance of conflict or inadvertent changes.

  * provides the ips and azs that will be used by the network. Setting
    `params.ocfp-instances` will allow a subset of these ips.

  * Provide a generated cloud config to specify the above values that will be
    uploaded prior to deployment.

- `stackit` - Configures the deployment for the STACKIT cloud platform. This
  sets up the appropriate VM extensions and other STACKIT-specific settings.

- `azure` - Configures the deployment for Microsoft Azure, including setting
  up availability sets for the OpenBAO nodes.

## Base Parameters

- `openbao_domain` - Domain for OpenBAO certs to be signed for. Defaults to
  `openbao.bosh`.

- `openbao_network` - What network to deploy OpenBAO into. This network must
  be defined in your cloud config. Defaults to `openbao`.

- `openbao_vm_type` - What type of VM to deploy. This type must exist in your
  cloud config. Defaults to `default`.

- `openbao_disk_type` - The persistent disk type that the OpenBAO VMs will
  use. This vm type must exist in your cloud config. Defaults to `default`.

- `ips` - If you want to have full control over the number of instances to
  deploy to, just specify the IPs of those instances in a list and those will
  be what is deployed. Specify one IP, one vm will be used; specify 5, 5 vms
  will be created.

  NOTE: Fault tolerance of Raft-based consensus is N/2+1 remaining instances,
  so even-counts of VMs don't increase fault tolerance over one less than that
  count. For this reason, it is recommended that odd-counts of IPs are
  specified.

- `availability_zones` - What BOSH HA availability zones to deploy OpenBAO
  across. The chosen network must have at least one subnet in each of the
  listed zones, and the zones themselves must be defined in your cloud config.
  Defaults to `z1`, `z2`, and `z3`.

- `stemcell_os` - The operating system stemcell you want to deploy on.
  (default: `ubuntu-jammy`)

- `stemcell_version` - The specific version of the stemcell you want to deploy
  on. (default: `latest`)

- `ui` - If true, the OpenBAO Web UI will be turned on. Defaults to `true`.

- `log_level` - OpenBAO server log level. Defaults to `info`.

- `default_lease_ttl` - Default lease TTL for secrets. Defaults to `768h`.

- `max_lease_ttl` - Maximum lease TTL for secrets. Defaults to `768h`.

- `auxiliary_vault` - If deploying a second vault after the primary Genesis
  Vault was deployed. This prevents automatic unsealing after a deploy, as
  Genesis will attempt to unseal the new instance with the seal keys of the
  primary Genesis Vault. Defaults to `false`.

- `azure_availability_set` - In Microsoft Azure, this parameter names the
  availability set to deploy the OpenBAO nodes across. This parameter does
  not have any effect on other platforms. Defaults to `openbao_as`.

## OCFP-specific Parameters

- `ocfp-subnet-prefix` - specifies the subnet prefix for subnets that will be
  used to populate the network components on this environment. Defaults to
  `ocfp`.

- `ocfp-instances` - specify how many instances to use out of the total
  provided by the OCFP configuration.

## Cloud Configuration

By default, OpenBAO uses the following VM types/networks/disk pools from your
cloud config. Feel free to override them in your environment, if you would
rather they use entities already existing in your cloud config:

```
params:
  openbao_network:   openbao
  openbao_disk_type: default  # should be at least 1GB
  openbao_vm_type:   default  # VMs should have at least 1 CPU, and 1GB of memory
```

## Available Addons

- `init` - Initializes the OpenBAO cluster. This can only be run once, as it
  formats the storage backend and generates seal keys for protecting the
  cluster.

- `target` - Sets up a `safe` target for this OpenBAO cluster, named after
  the environment, and prompts for authentication. Takes an optional
  `<auth-type>` argument to specify the auth method, such as `github` or
  `userpass`, defaulting to `token` if not provided.

- `status` - Determine OpenBAO status: health, availability, and sealed /
  unsealed state.

- `seal` - Seals the OpenBAO cluster.

- `unseal` - Unseals the OpenBAO cluster.

## Certificate Management

The kit manages three certificate chains:

- **CA** - Root certificate authority, valid for 10 years. Signs all other
  certificates.

- **Peer** - Used for Raft peer-to-peer mTLS communication between cluster
  nodes. Valid for 10 years. SANs include `openbao_raft_peer` and `127.0.0.1`.

- **Vault** - Client-facing TLS certificate for API access. Valid for 2 years.
  SANs include the configured domain and `127.0.0.1`.

## Architecture

OpenBAO uses integrated Raft storage for high availability and data
persistence. Each node in the cluster participates in Raft consensus:

- One node is elected leader and handles all write operations
- Follower nodes replicate data from the leader
- A quorum of N/2+1 nodes must be available for the cluster to accept writes
- Peer discovery uses BOSH links for automatic cluster formation
- All inter-node communication is secured with mTLS using the peer certificate

BPM (BOSH Process Manager) is used to manage the OpenBAO process lifecycle,
providing cgroup isolation, consistent logging, and clean process management.

## Examples

To use custom cloud config types:

```
---
kit:
  name: openbao
  version: 1.0.0

genesis:
  env: acme-us-east-1-prod

params:
  openbao_network:   credstore
  openbao_disk_type: encrypted
  openbao_vm_type:   std.small.1c.2gb
```

To deploy on Azure:

```
---
kit:
  name: openbao
  version: 1.0.0

genesis:
  env: azure-eu-prod

params:
  azure_availability_set: openbao-as
```

To deploy using OCFP configurations:

```
---
kit:
  name: openbao
  version: 1.0.0
  features:
    - ocfp

genesis:
  env: ocfp-us-prod

params:
  ocfp-instances: 3
```

## Caveats

Each OpenBAO installation will have its own seal keys and initial root token.
The natural inclination will be to store these _in_ the vault, but that's a
bad idea. Use an external password manager instead, like 1password or LastPass.

Every time OpenBAO is deployed, whether to pick up configuration changes or to
upgrade the operating system (stemcell), the cluster will protectively seal
itself. An operator **must** intervene to unseal it, or it will remain
inaccessible.

## History

Version 1.0.0 is the initial release of the OpenBAO Genesis Kit, providing
Raft-based storage, BPM process management, and full compatibility with the
